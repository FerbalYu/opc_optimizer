# OPC Optimizer 后续路线图

> 更新时间：2026-03-25
> 当前基线：Step 23 已完成，Web UI 审核流已实现首版

---

## 当前定位

OPC 现在已经具备：

- 项目画像与 SKILL 辅助规划
- 结构化每轮合同
- 执行范围约束
- 构建 / 测试 / UI 验证
- 低价值回合识别
- 真实 diff 证据复盘
- Web UI 计划审核首版

后续路线不再是“把能力做多”，而是把下面三件事继续做深：

1. 高风险逻辑先验证再改
2. 每轮更快、更聚焦、更省 token
3. Web UI 审核流变成稳定可依赖的主交互模式

---

## Step 24: Verification-First Mode

### 目标

当目标涉及算法、规则引擎、财务计算、领域判定逻辑时，优先补最小验证，再做核心修改。

### 为什么要做

当前 19 到 23 步已经能发现“低价值回合”，但还不能完全避免“有勇气改、没把握验”的问题。高风险逻辑如果没有最小验证样例，改动再漂亮也可能不可靠。

### 计划内容

- 在 `plan_node` 中加入高风险逻辑识别
- 如果目标文件缺少测试或可执行验证：
  - 本轮优先任务改成“建立基线验证”
  - 而不是立即大改核心逻辑
- `test_node` 增加对新基线样例的执行与记录

### 验收标准

- 高风险改动前，至少有一条可执行验证路径
- 回合报告能明确指出“先建验证”还是“已可安全改逻辑”

## Step 25: Efficiency Upgrade Pass

### 目标

减少无效 token、无效上下文和无效 round。

### 计划内容

- 让 `round_contract.target_files` 更小更硬
- 在计划和执行阶段都进一步压缩候选文件集
- 对已知低信号文件提前跳过
- 缓存常见失败模式与低质量提示模板
- 把“整项目扫描”严格限制为首轮或必要重规划时才做
- 减少 review 阶段重复传输的历史信息

### 验收标准

- 单轮上下文更短
- 偏题回合更少
- 同样轮数下交付的价值更高

---

## Step 26: Web UI 审核流完成态

### 目标

把当前首版任务审核流做成正式工作模式。

### 已有基础

- landing 已支持 `skip_plan_review`
- `plan_node` 已支持：
  - 任务列表生成
  - 用户批准
  - 用户驳回后重规划
- dashboard 已支持：
  - 任务条
  - 任务详情弹窗
  - 审核弹窗

### 待补内容

- 完整端到端联调
- 配置页 / 运行页状态一致性
- 审核结果后的任务状态表现打磨
- 用户整批否决后的反馈文案和下一批任务差异化更强
- 更清晰的“本轮实际执行了哪几个任务”可视化

### 验收标准

- 不审核模式可稳定全自动运行
- 审核模式下可稳定完成：
  - 选择部分任务后继续
  - 全部否决后换一批
  - 点击任务查看详情

## 远期方向

当 24-26 稳定后，再考虑更大层级的升级：

- Worker 模式，把 OPC 分成“规划监工”和“执行工人”
- 更强的真实操作验证
- 面向不同项目类型的专用审核策略
