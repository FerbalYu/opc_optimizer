# 基于 gstack 的 OPC Optimizer 改造方案与实施计划

## 1. 目标与范围

### 1.1 改造目标

- 将 `opc_optimizer` 从“单一优化循环引擎”升级为“可路由、可组合、可治理的技能化工作流系统”。
- 引入类似 `gstack` 的方法学分层：`技能定义`、`技能路由`、`统一前置流程`、`质量门禁`、`文档自动同步`。
- 保持现有核心价值（`LangGraph` 多轮优化 + 合同约束 + 自动验证）不倒退。

### 1.2 本次计划边界

- 先做架构与工程化改造，不一次性复制 `gstack` 全部业务技能。
- 优先改造“技能系统与流程编排底座”，再逐步迁移/新增技能能力。
- 保持 Python 主体实现，借鉴 `gstack` 的设计理念和机制，而非语言层面完全一致。

## 2. 现状对比（gstack vs opc_optimizer）

### 2.1 opc_optimizer 现状（已具备）

- 已有稳定主循环：`task_router -> plan -> execute -> test -> archive -> report -> interact`。
- 已有强约束机制：`round_contract`、目标文件白名单、`.bak` 回滚、构建/测试验证。
- 已有 Web UI 交互与事件推送、项目画像、上下文裁剪、技能文档注入（`opcskills/*.md`）。

### 2.2 gstack 可借鉴能力（核心）

- 技能即产品：每个技能都有明确职责边界和标准化执行协议。
- 模板化生成：`SKILL.md.tmpl -> 生成脚本 -> SKILL.md`，防止文档和实现漂移。
- 统一 preamble：更新检查、会话态、策略配置、行为约束前置执行。
- 路由机制：根据用户意图自动分派到正确技能，而不是单一路径硬跑。
- 多 host 适配：同一技能体系可适配不同 Agent 运行环境。
- 质量体系：静态校验 + E2E + 文档新鲜度校验（CI 可控）。

### 2.3 当前差距

- `opcskills` 仍是“提示片段库”，不是“可执行技能单元”。
- 缺少统一技能前置协议（配置读取、模式判断、日志与治理动作）。
- 缺少“技能路由层 + 技能生命周期管理层”。
- 缺少“模板驱动的技能文档生成/校验”机制。
- 缺少针对“技能级别”的测试矩阵与版本管理策略。

## 3. 总体改造原则

- 最小侵入：优先新增模块，不大改现有 `plan/execute/test` 核心逻辑。
- 分层落地：先底座（模型与路由）再迁移（现有节点技能化）再扩展（新技能）。
- 双轨运行：保留现有优化主流程开关，确保可回退。
- 可观测优先：每个阶段都先补齐事件/日志，再扩功能。
- 文档先行：每完成一个阶段同步更新架构文档与使用文档。

## 4. 目标架构（改造后）

### 4.1 新增分层

- `skill_registry`：技能注册表（元数据、入口、依赖、执行条件、风险级别）。
- `skill_router`：意图/上下文路由器（将请求映射到技能链路）。
- `skill_runtime`：技能执行运行时（统一 preamble、上下文、审计、容错、收敛）。
- `skill_docs_pipeline`：模板生成与校验（`*.tmpl -> *.md`）。
- `workflow_bridge`：把现有节点流程封装为一个或多个基础技能。

### 4.2 与现有节点关系

- 短期：节点仍存在，新增“技能封装器”调用现有节点函数。
- 中期：路由器可按目标选择“传统优化环”或“技能化流程”。
- 长期：节点演进为标准技能，图编排成为技能 DAG，而非固定线性链。

## 5. 分阶段实施计划（一步一步）

## Phase 0：基线冻结与可观测补齐（1-2 天）

- 建立基线：记录当前测试通过率、关键耗时、典型输入输出样本。
- 补齐观测字段：技能名、路由决策、阶段耗时、失败原因分类。
- 增加运行模式标识：`legacy_mode` / `skill_mode`。

验收标准：

- 不改业务行为，现有测试全绿。
- 新增日志不影响现有输出协议（CLI/Web UI）。

## Phase 1：技能模型与注册中心（2-3 天）

- 新增 `utils/skill_registry.py`：
- 定义 `SkillSpec`（name、description、entrypoint、inputs、outputs、safety_level、enabled）。
- 提供注册/发现/查询 API，支持后续插件化扩展。
- 将现有 `opcskills` 内容从“纯文本策略”映射为“策略资源”，挂到 `SkillSpec.resources`。

验收标准：

- 可以列出可用技能清单（最少含 `plan/execute/test/report` 封装技能）。
- 注册中心可在单测中验证冲突检测与禁用逻辑。

## Phase 2：技能路由器与统一 preamble（3-5 天）

- 新增 `nodes/skill_router.py` 或 `utils/skill_router.py`：
- 根据目标、项目画像、历史失败类型做技能选择。
- 引入统一 preamble：
- 读取运行配置（自动模式、风险模式、是否允许破坏性操作）。
- 注入统一执行上下文（session_id、round_id、project_profile 摘要）。
- 兼容现有 `task_router`，初期使用可配置开关切换。

验收标准：

- 路由器能稳定输出一个技能执行计划（含顺序与跳转条件）。
- 当路由失败时可降级回现有线性流程。

## Phase 3：把现有节点“技能化封装”（4-6 天）

- 为 `plan/execute/test/interact/report` 增加技能包装层，不重写内部算法。
- 统一技能输入输出契约（避免节点内部字段耦合扩散）。
- 每个技能增加：
- 前置校验（输入状态完整性）。
- 后置审计（变更摘要、风险标签、失败归因）。

验收标准：

- `skill_mode` 下至少跑通与当前等价的一轮流程。
- 结果报告可对照 `legacy_mode`，关键行为一致。

## Phase 4：技能文档模板化与自动生成（2-4 天）

- 新增 `skills/*.md.tmpl`（或复用 `opcskills` 目录并扩展元数据）。
- 新增生成脚本（Python 实现，借鉴 `gstack/gen-skill-docs.ts` 思路）：
- 解析模板占位符。
- 从代码元数据生成命令/参数/限制段落。
- 输出最终技能文档。
- 新增 `--check-skill-docs` 校验命令，接入 CI。

验收标准：

- 文档可一键生成。
- 当技能接口变更但文档未更新时，CI 能失败并给出定位信息。

## Phase 5：质量门禁与测试矩阵升级（3-5 天）

- 新增技能级测试：
- 单元：注册、路由、preamble、契约校验。
- 集成：技能链路执行（至少 3 条代表路径）。
- 回归：`legacy_mode` 对照测试。
- 引入“变更影响面校验”：技能修改是否越界到非目标文件。

验收标准：

- 新增测试稳定通过，且现有测试不回归。
- 至少覆盖一个失败自修复场景和一个回滚场景。

## Phase 6：逐步启用与迁移（2-3 天）

- 默认仍用 `legacy_mode`，对特定项目/参数灰度启用 `skill_mode`。
- 收集 1-2 周运行数据后再切换默认模式。
- 保留紧急回退开关（环境变量 + CLI 参数双通道）。

验收标准：

- 线上/本地运行无明显稳定性退化。
- 出现异常可在一次运行内快速回切。

## 6. 首批文件级改造清单（建议）

- 新增：
- `utils/skill_registry.py`
- `utils/skill_router.py`
- `utils/skill_runtime.py`
- `skills/`（或扩展 `opcskills/` 为模板目录）
- `scripts/gen_skill_docs.py`
- `tests/test_skill_registry.py`
- `tests/test_skill_router.py`
- `tests/test_skill_runtime.py`

- 修改（最小侵入）：
- `graph.py`（入口增加 `skill_mode` 路由分支）
- `state.py`（新增技能运行态字段）
- `main.py`（新增开关与模式输出）
- `nodes/task_router.py`（与技能路由协同）
- `utils/skill_loader.py`（兼容新模板元数据）

## 7. 风险与回滚策略

- 风险：状态字段膨胀导致节点兼容性问题。
- 策略：新字段统一放 `skill_runtime` 子结构，避免平铺污染。

- 风险：路由错误导致流程不收敛。
- 策略：所有路由失败自动降级 `legacy_mode`，并写入原因。

- 风险：文档生成机制引入维护负担。
- 策略：先支持最小占位符集合，不追求一次覆盖全部动态内容。

## 8. 执行顺序建议（从下一步开始）

1. 先落地 `Phase 0`，建立基线与观测字段。
2. 再做 `Phase 1`：只引入注册中心，不动业务路径。
3. 再做 `Phase 2`：引入路由器，但默认关闭。
4. 接着做 `Phase 3`：节点技能化封装，形成可跑通最小闭环。
5. 最后推进 `Phase 4~6`，完成文档自动化、测试矩阵和灰度切换。

---

## 9. 当前结论

`gstack` 最值得迁移的不是“具体命令集合”，而是“技能化生产流程的工程化骨架”。  
建议按本计划逐步改造 `opc_optimizer`，优先构建可路由、可审计、可验证的技能底座，再迁移能力细节。
